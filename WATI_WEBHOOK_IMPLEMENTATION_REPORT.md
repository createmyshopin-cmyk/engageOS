## 1. Architecture

The inbound layer is a thin, verify-fast / process-async pipeline:

```
WATI  ──POST /api/webhooks/wati?token=<secret>──▶  route.ts
                                                     │
                        1. readToken() (query ?token / ?secret / Bearer)
                        2. resolveWatiTenant(token)  ── UNIQUE webhook_token ─▶ business
                        3. parse body (defensive)
                        4. HTTP 200 ACK  ◀── returned immediately
                        5. after(() => processWatiWebhook(integration, body))
                                                     │
                    ┌────────────────────────────────┴───────────────────┐
                    ▼                                                     ▼
         claimWatiWebhookDelivery()                            (duplicate → stop)
         (idempotency ledger, UNIQUE)
                    │ won the claim
                    ▼
         mapEvent() → coupon status + campaign_events type
                    │
          ┌─────────┴──────────┐
          ▼                    ▼
  applyCouponReceipt()   recordCampaignEvent()
  (forward-only guard)   (immutable lifecycle log, channel=wati)
```

## 2. Endpoint

| Method | Path | Purpose |
|--------|------|---------|
| `GET`  | `/api/webhooks/wati?token=…` | Health/verification probe. Returns `200 {ok:true,status:"ready"}` for a valid token, `401` otherwise. Lets a merchant confirm wiring; leaks no tenant data. |
| `POST` | `/api/webhooks/wati?token=…` | Receives events. Verifies → `200` ACK → async processing. Malformed body is still `200` (see §10). Invalid token → `401`. |

`runtime = "nodejs"`, `maxDuration = 60`. The async work runs inside `after()` from `next/server`, which extends the serverless invocation lifetime via `waitUntil` so the coupon/receipt writes complete after the response is flushed (critical on Vercel, where a detached promise would be frozen).

## 3. Supported Events

| WATI `eventType` | Internal status | `campaign_events.event_type` | Coupon effect |
|------------------|-----------------|------------------------------|---------------|
| `sentMessage`, `sentMessageSENT` | Sent | `whatsapp.sent` | `wa_status → sent`, `wa_sent_at` |
| `sentMessageDELIVERED` | Delivered | `whatsapp.delivered` | `wa_status → delivered`, `wa_delivered_at` |
| `sentMessageREAD` | Read | `whatsapp.read` | `wa_status → read`, `wa_read_at` |
| `templateMessageFailed` | Failed | `whatsapp.failed` | `wa_status → failed`, `wa_failed_at`, `wa_failed_reason` |
| `messageReceived` | Received | `whatsapp.received` | none (inbound reply logged) |
| `templateStatusUpdated` | — | none (claimed + logged as no-op) | none |
| *unknown* | — | none | none — never crashes |

Unknown event types fall through `mapEvent()`'s `default` and are acknowledged without side effects.

## 4. Webhook Verification

WATI provides **no native HMAC signature** (confirmed against the WATI spec — unlike Meta's `X-Hub-Signature-256`). Verification is therefore a **high-entropy per-tenant bearer token**:

- `wati_integrations.webhook_token` — 64 hex chars (two `gen_random_uuid()` values concatenated), `NOT NULL`, `UNIQUE`, auto-generated by column default and backfilled for existing rows.
- Carried in the callback URL the merchant registers: `…/api/webhooks/wati?token=<secret>`. Also accepted via `Authorization: Bearer <secret>` or the legacy `?secret=` alias.
- The token doubles as the **tenant key**: because the column is `UNIQUE`, a token resolves to exactly one business or to nobody. There is no tenant id in the payload to spoof.
- A missing/short/unknown token → `401`, no processing, no disclosure of whether the token was close.

The token is never logged and never returned to the browser except as part of the merchant's own webhook URL on their authenticated settings screen.

---

## 5. Idempotency Strategy

WATI retries a non-200 (or timed-out) delivery **up to 144 times at 10-minute intervals (24 hours)**. Every event is claimed before any side effect:

- Ledger table `wati_webhook_deliveries` with `UNIQUE(business_id, dedup_key)`.
- `dedup_key = eventType : (whatsappMessageId | localMessageId | timestamp) : statusString`. Distinct status transitions on the same message each process once; an identical re-delivery loses the insert race (`23505`) and returns `"duplicate"`.
- The claim happens **inside** `processWatiWebhook`, before the coupon update or event log, so no `campaign_events`, coupon receipt, or analytics increment can ever be doubled by a retry.
- The coupon update itself is additionally guarded by a compare-and-set (`.eq("wa_status", expected)`) so two concurrent receipts cannot both apply.

## 6. Database Mapping

**`coupons`** (delivery receipts, additive columns): `wa_sent_at`, `wa_delivered_at`, `wa_read_at`, `wa_failed_at`, `wa_failed_reason`. `wa_status` check widened from `('pending','sent','failed')` to `('pending','sent','delivered','read','failed')` — strictly additive; all previously legal values remain legal, and the existing pending partial index and migration-0009 rollups are unaffected.

**`campaign_events`** (immutable lifecycle log): receipts append `whatsapp.sent/delivered/read/failed`; inbound replies append the new `whatsapp.received`. All carry `metadata.channel = "wati"`, `metadata.source = "webhook"`, so the existing WATI analytics route (`/api/m/wati/analytics`, which already counts `whatsapp.delivered` and `whatsapp.read`) and the merchant dashboard light up with zero further wiring. `wacrm` (Meta) sends stay uncounted because they lack `channel=wati`.

**`wati_webhook_deliveries`** (new): idempotency ledger only.

Coupon linkage: the outbound sender sets `broadcastName = coupon_${code}`; `extractCouponCode()` recovers the code from WATI's echoed `localMessageId` via `/coupon(?:_delivery)?_([A-Za-z0-9-]+)/`, tolerating an optional `_delivery` segment. Participation and inbound messages have no coupon code and correctly skip the coupon update while still logging the event.

---

## 7. Tenant Mapping

One webhook → exactly one business, enforced structurally:
1. Token is `UNIQUE` at the database level.
2. Every query in the processing path is scoped by `integration.business_id` (coupon lookup, ledger claim, event insert).
3. There is no tenant identifier in the payload that could redirect a webhook to another tenant.

A token that resolves to no row is treated as an unknown tenant → `401`, no work.

## 8. Performance

- **Fast ACK:** verification + parse are the only synchronous work before `200`. Idempotency claim, coupon update, and event insert run in `after()`.
- **O(1) tenant resolution** via the unique token index.
- **Bounded writes:** at most one ledger insert, one coupon update, one event insert per unique event; retries short-circuit at the claim.
- **Retention:** `wati_webhook_deliveries.created_at` is indexed so rows older than the 24h retry window can be swept by a routine cleanup (rows are only needed for the retry horizon).

---

## 9. Security

- Per-tenant secret token; constant-effort `401` on any invalid/unknown/short token.
- **Logging is lifecycle-only.** Emitted lines: `received + verified`, `processed <event> → <type>`, `duplicate ignored`, `no-op event acknowledged`, `malformed payload ignored`, `rejected: invalid or unknown token`, `tenant resolve failed`. **Never logged:** API tokens, the webhook secret, phone numbers, sender names, or message text (`text` is parsed off the payload but never read into a receipt or a log).
- Payloads are never trusted: every field goes through `str()` coercion; `parseWatiEvent` returns `null` on anything without a valid `eventType`.
- Service-role-only DB access; `wati_webhook_deliveries` has RLS enabled default-deny with grants revoked from `anon`/`authenticated`, matching the rest of the schema.

- `npx eslint` on all added/changed files → **0 warnings/errors**.

Behavioural walk-throughs traced against the code:
- **Happy path** SENT→DELIVERED→READ on a `coupon_ABC-123` message: each advances `wa_status` one rung, stamps the matching timestamp, appends one `campaign_events` row. ✔
- **Duplicate** (same event re-sent): claim loses on `23505` → `"duplicate"`, no second write. ✔

- **Failure**: `templateMessageFailed` from `sent` → `wa_status=failed`, `wa_failed_reason=wati_<code>`. From `read` → guard blocks regression. ✔
- **Unknown tenant / bad token** → `401`, no processing. ✔
- **Malformed body / unknown event / missing coupon / deleted campaign** → acknowledged, no crash (see below). ✔

## 11. Remaining Limitations

- **Live WATI delivery not exercised here** — verification is static + trace-based; a staging round-trip from the WATI dashboard is recommended before GA.
- **`messageReceived` is logged, not conversationalised** — inbound replies append a `whatsapp.received` event but do not (by design / scope constraint) create a conversation thread or auto-reply. Keyword bots remain WATI-native.
- **Ledger retention** relies on an external periodic cleanup; no cron is bundled (rows are harmless if retained, just unbounded growth over time).
- **`localMessageId` format coupling** — coupon linkage depends on WATI echoing the `coupon_<code>` broadcast name. If a merchant sends coupon templates through a path other than EngageOS's sender, those receipts won't map to a coupon (they are still logged).
- **`sentMessage` and `sentMessageSENT`** both map to `sent`; if WATI emits both for one message they are distinct dedup keys but the second is a no-op transition (already `sent`), so no double count.
